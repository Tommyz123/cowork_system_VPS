# 系统复盘流程

> 目标：找出反复出现的摩擦，把靠AI自律的规则变成结构性保障。

---

## 步骤0：Preamble（读取上次状态）

```bash
cat ~/.claude/skills/系统复盘/state.md 2>/dev/null || echo "（首次运行，无历史状态）"
```

- **有 state.md** → 加载上次发现的 known_friction 列表；本次分析时对每个摩擦标注：`已知`（上次有，本次还在）/ `已修`（上次有，本次消失）/ `新增`（本次首次出现）/ `复发`（上次已修，本次又出现）
- **无 state.md** → 全量分析，当作首次运行

---

## 步骤1：读取 friction_log.md

```bash
cat /root/cowork/friction_log.md
```

---

## 步骤2：归类统计

按类型分组统计：
- 规则模糊 / 规则冲突
- 行为被纠正
- 工具限制
- 流程缺失

输出各类型条数，标注哪些是**未解决**状态。

---

## 步骤3：检查复发

找出同一场景出现 ≥2次 的摩擦，标记为"复发问题"——这些是最需要结构性解决的。

---

## 步骤3.5：系统健康阈值检查

检查以下条件，达到则在报告中标注 ⚠️：
- legal_library 文件 > 30个 → 建议升级语义搜索
- 活跃项目 > 5个 → 建议建项目健康仪表盘
- BACKLOG 未实现条目 > 8条 → 建议排优先级
- friction_log.md 中"规则模糊/规则冲突/行为被纠正"类型累计 > 10条 → 建议将对应规则代码化
- friction_log.md 未归档条目 > 5条 → 提醒立即归档

---

## 步骤4：输出报告

**信心评分规则**：每个摩擦条目在输出前先打分（1-10）：
- ≥7 → 完整输出（场景+处理方式+建议）
- 4-6 → 只列标题，不展开
- <4 → 静默跳过，不输出

```
📊 系统复盘报告
总条数：X | 未解决：X | 复发：X | 新增：X | 已修：X

🔴 复发/新增问题（≥7分，完整展开）：
- [场景] [新增/复发] 出现N次，当前状态：...

🟡 已知未解决问题（4-6分，仅标题）：
- [场景标题]...

🟢 已闭环 / 已修：
- X条（本次不展开）
```

---

## 步骤5：等主公确认后执行修改

针对复发问题，提出解决方案（Hook化 / Skill化 / 修改规则），等主公确认后再执行。

确认闭环的条目移至 `friction_log_archive.md`，`friction_log.md` 只保留活跃条目。

---

## 步骤6：写回状态文件

修改执行完毕后，更新 `~/.claude/skills/系统复盘/state.md`：

```bash
cat > ~/.claude/skills/系统复盘/state.md << 'EOF'
last_run: [YYYY-MM-DD]
score: [未解决X条 / 复发Y条 / 新增Z条]
known_friction:
  - [摩擦摘要] | 状态：已知/已修/复发/新增
  （逐条列出本次所有≥4分的条目）
EOF
```
