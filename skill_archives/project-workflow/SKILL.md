# 一键项目规划流程

自动执行完整流程：生成 → 审核 → 修复 → 直到全部通过。

```
需求文档 → PROJECT_PLAN → [审核循环] → TODOLIST → [审核循环] → 完成
```

---

## 阶段1: 生成项目计划书

### 1.1 读取输入文档

#### 必需：需求文档
按优先级搜索：requirements.md > README.md > 需求.txt > 需求文档.md > *.txt

#### 可选：解决方案文档
按优先级搜索：SOLUTION_BLUEPRINT.md > SOLUTION_BLUEPRINT.docx > 解决方案.md > 解决方案.docx

> **定位说明**：解决方案文档定义战略约束和边界限制。
> - 标记为 "Frozen" 的决策不可违反
> - 它约束"什么不能做"，而非定义功能需求
> - 生成的计划必须遵守这些约束
> - 如未找到，正常继续流程，不报错

### 1.2 判断项目阶段

| 阶段 | 条件 | 维度数 |
|------|------|-------|
| MVP期 | 代码<500行 或 模块≤2 | 9 |
| 扩展期 | 代码<2000行 | 11 |
| 成熟期 | 代码≥2000行 | 13 |

### 1.2.5 约束检查（如有解决方案文档）

如果读取到解决方案文档，在生成每个维度时：
1. 检查是否有相关的 Frozen 决策
2. 确保生成内容不违反冻结约束
3. 在维度内容中标注约束来源（如适用）

示例约束：
- 架构模式被冻结 → 维度3必须使用指定架构
- 技术栈被冻结 → 维度4必须使用指定技术
- 复杂度限制被冻结 → 不能引入超出限制的设计

### 1.3 生成维度内容

**9个基础维度（MVP必需）**：

| 维度 | 标题 | 必需内容 |
|------|------|---------|
| 1 | 项目概述 | 产品愿景、目标用户、核心价值、项目类型 |
| 2 | 需求分析 | 功能需求(P0/P1)、非功能需求、业务规则 |
| 3 | 系统架构 | 架构模式、架构图(含数据流)、SOLID符合度 |
| 4 | 技术栈选型 | 每项只写一个，备注列写替代方案 |
| 5 | 核心模块设计 | 每模块：职责、位置、接口、扩展点 |
| 6 | API接口文档 | 端点列表、请求/响应格式 |
| 7 | 数据模型设计 | 表定义、字段说明、数据流 |
| 8 | 项目结构 | 目录树、目录说明 |
| 9 | 测试与验证策略 | 测试框架、测试类型、验收标准、测试命令 |

扩展期增加：维度10(部署安装)、11(使用指南)
成熟期增加：维度12(成本估算)、13(常见问题)

### 1.4 强制规则

1. **维度9必须是"测试与验证策略"**（不是部署）
2. 技术选型只写一个，禁止"A / B"写法
3. 核心模块 ≥ 3 个
4. 必须有"初步可行性评估"章节（4项评分）
5. TODO标记 ≤ 10 个

### 1.5 输出 PROJECT_PLAN_v1.md

## 阶段1.5: 审核项目计划书

### ⚠️ 审核铁律

1. **必须输出审核报告** — 不能跳过，不能只说"通过"
2. **每项检查必须引用原文** — 禁止无证据打 ✅
3. **找不到证据 = 不通过** — 不能假设存在
4. **禁止自动给高分** — 必须逐项扣分后计算

### 结构规则检查（P0级，任一失败直接不通过）

用 Grep 实际搜索验证，输出证据：

| 检查项 | 检查方法 | 证据要求 |
|--------|---------|---------|
| 维度数量 | 搜索 `## 1.` 到 `## 9.`（或11/13）计数 | 输出："找到维度标题：1,2,3..." |
| 维度9标题 | 搜索 `## 9.` 后的文字 | 输出："维度9标题为：{实际标题}" |
| 可行性评估 | 搜索 "初步可行性评估" 或 "可行性评估" | 输出："找到章节：{行号}" 或 "未找到" |
| 生成信息 | 搜索 "生成信息" 或 "生成时间" | 输出："找到章节：{行号}" 或 "未找到" |

**任何一项"未找到" → 直接 P0，停止审核，要求修复**

### 内容规则检查（P1级）

| 检查项 | 检查方法 | 扣分 |
|--------|---------|------|
| 技术选型有"/"或"或" | 搜索维度4中的 `/` `或` `OR` | 有 → -15分 |
| 核心模块不足 | 数维度5中 `### 模块` 数量 | <3个 → -10分 |
| TODO过多 | 搜索 `TODO` `待定` `TBD` | >10个 → -5分 |
| 占位符过多 | 搜索 `{` `}` `xxx` `...` | >20处 → -10分 |

### 7维度评分（必须逐项打分，给出理由）

**输出格式（必须）**：

```
### 审核评分

| 维度 | 满分 | 得分 | 扣分原因 |
|------|------|------|----------|
| 需求对齐度 | 15 | {X} | {具体原因或"无"} |
| 架构设计质量 | 20 | {X} | {具体原因或"无"} |
| 技术方案完整性 | 15 | {X} | {具体原因或"无"} |
| 命名一致性 | 10 | {X} | {具体原因或"无"} |
| 可执行性 | 20 | {X} | {具体原因或"无"} |
| 测试验证完备性 | 10 | {X} | {具体原因或"无"} |
| 文档质量 | 10 | {X} | {具体原因或"无"} |
| **总分** | 100 | {SUM} | - |
```

**扣分标准**：

| 维度 | 扣分情况 |
|------|---------|
| 需求对齐度 | P0功能未覆盖: -5/个，非功能需求无方案: -3 |
| 架构设计质量 | 无架构图: -10，无数据流: -5，无SOLID说明: -5 |
| 技术方案完整性 | 模糊表述("可以用X"): -3/处，占位符: -1/处 |
| 命名一致性 | 前后命名不一致: -3/处 |
| 可执行性 | 实现步骤不具体: -5，模块协作不清晰: -5 |
| 测试验证完备性 | 无测试框架: -5，无验收标准: -5 |
| 文档质量 | 缺少维度: -3/个，TODO>10: -3 |

### 评分判定

- ≥90分 → 输出"✅ PROJECT_PLAN 审核通过"，进入阶段2
- <90分 → 输出问题列表，进入阶段1.8修复

---

## 阶段1.8: 修复 PROJECT_PLAN

### 修复原则

1. **只修审核报告列出的问题** — 不改动已通过的内容
2. **对照生成规则补齐** — 审核说缺什么，回看阶段1的规则补什么
3. **修复后必须重审** — 不能自己说修好了就过

### 修复流程

```
1. 读取审核报告，提取 P0/P1 问题列表
2. 对每个问题：
   a. 定位问题类型（结构缺失？内容不足？格式错误？）
   b. 回看阶段1对应规则
   c. 按规则补齐/修正
3. 生成 PROJECT_PLAN_v{N+1}.md
4. 返回阶段1.5重新审核
```

### 常见问题修复

| 问题 | 修复方法 |
|------|---------|
| 维度9标题错误 | 将当前维度9改为维度10，新增维度9"测试与验证策略" |
| 缺可行性评估 | 在附录添加4项评分表格 |
| 技术选型有"/" | 选一个，另一个放备注列 |
| 模块不足3个 | 从需求中拆分出更多模块 |
| TODO过多 | 根据需求文档填充具体内容 |

### 重试限制

最多修复 3 次。3 次后仍 <90 分 → 停止，输出：
```
⚠️ 修复3次仍未通过，请检查需求文档是否完整，或手动修改 PROJECT_PLAN
```

---

## 阶段2: 生成任务清单

**前置：PROJECT_PLAN 已通过（≥90分）**

### 2.1 提取模块

从 PROJECT_PLAN 维度5 提取所有核心模块

### 2.2 分析依赖

构建依赖图，拓扑排序：
- 无依赖的模块排最前
- 有依赖的模块在依赖项之后
- 检测到循环依赖 → 报错

### 2.3 生成 Task

每个 Task **必须**包含 6 部分：

```
**功能要求**: （≥3条）
1. {功能点1}
2. {功能点2}
3. {功能点3}

**实现位置**:
- 文件: src/{path}/{module}.py
- 测试: tests/test_{module}.py

**代码审核**:
pylint {模块路径}
bandit -r {模块路径}

**测试验证**:
pytest {测试文件} -v
测试用例:
- test_{功能1}
- test_{功能2}

**验收标准**:
- [ ] {标准1}
- [ ] {标准2}

**完成后必须执行**:
1. 确认代码审核通过
2. 确认测试通过
3. 更新进度
4. 检查下一个Task前置条件
```

### 2.4 必需章节

1. 进度追踪（显示0%）
2. 任务依赖关系（ASCII图）
3. 任务详情
4. 首次执行指令
5. 工具说明
6. 注意事项

### 2.5 初始状态规则

1. 进度 = 0%
2. 所有 Task 状态 = "未开始"
3. 所有 checkbox 未勾选 `[ ]`

### 2.6 输出 TODOLIST.md

---

## 阶段2.5: 审核任务清单

### ⚠️ 审核铁律（同上）

1. **必须输出审核报告**
2. **每项检查必须引用原文**
3. **找不到证据 = 不通过**
4. **禁止自动给高分**

### 结构规则检查（P0级）

用 Grep 搜索验证 6 个必需章节：

| 章节 | 搜索关键词 | 证据要求 |
|------|-----------|---------|
| 进度追踪 | "进度追踪" "进度" | 输出行号或"未找到" |
| 任务依赖关系 | "依赖关系" "依赖" | 输出行号或"未找到" |
| 任务详情 | "任务详情" "Task" | 输出行号或"未找到" |
| 首次执行指令 | "首次执行" "初始化" | 输出行号或"未找到" |
| 工具说明 | "工具说明" "工具" | 输出行号或"未找到" |
| 注意事项 | "注意事项" "注意" | 输出行号或"未找到" |

**任何一项"未找到" → P0，停止审核**

### Task 完整性检查（P0级，逐个验证）

**必须对每个 Task 单独检查，输出表格**：

```
### Task 完整性检查

| Task | 功能要求≥3 | 实现位置 | 代码审核 | 测试验证 | 验收标准 | 完成后执行 |
|------|-----------|---------|---------|---------|---------|-----------|
| Task 1 | ✅ "1.xx 2.xx 3.xx" | ✅ "src/..." | ✅ "pylint..." | ✅ "pytest..." | ✅ | ✅ |
| Task 2 | ❌ 只有2条 | ✅ | ❌ 未找到 | ✅ | ✅ | ✅ |
| ... | | | | | | |
```

**任何 Task 有 ❌ → P0，列出具体缺失**

### 初始状态检查（P0级）

| 检查项 | 检查方法 | 证据 |
|--------|---------|------|
| 进度=0% | 搜索进度数字 | 输出："进度显示为 {X}%" |
| 状态=未开始 | 搜索"已完成""进行中" | 输出："找到{N}个非'未开始'状态" |
| checkbox未勾选 | 搜索 `[x]` `[X]` | 输出："找到{N}个已勾选" |

**进度≠0% 或 有已勾选 → P0**

### 6维度评分

**输出格式（必须）**：

```
### 审核评分

| 维度 | 满分 | 得分 | 扣分原因 |
|------|------|------|----------|
| 三方对齐度 | 20 | {X} | {原因} |
| Task依赖正确性 | 20 | {X} | {原因} |
| Task粒度合理性 | 15 | {X} | {原因} |
| Task完整性 | 15 | {X} | {原因} |
| 质量保障可行性 | 15 | {X} | {原因} |
| 可执行性 | 15 | {X} | {原因} |
| **总分** | 100 | {SUM} | - |
```

**扣分标准**：

| 维度 | 扣分情况 |
|------|---------|
| 三方对齐度 | 需求未覆盖: -5/个，模块遗漏: -5/个 |
| Task依赖正确性 | 顺序错误: -5，循环依赖: -10，引用无效: -5 |
| Task粒度合理性 | Task过大(>8小时): -3，过小(<1小时): -2 |
| Task完整性 | 缺少部分: -3/个，内容空洞: -2/处 |
| 质量保障可行性 | 命令错误: -5，工具不匹配: -3 |
| 可执行性 | 步骤不清晰: -5，缺少前置条件: -3 |

### 评分判定

- ≥90分 → 输出"✅ TODOLIST 审核通过，可以开始开发"
- <90分 → 输出问题列表，进入阶段2.8修复

## 阶段2.8: 修复 TODOLIST

### 修复原则

1. **只修审核报告列出的问题** — 不改动已通过的内容
2. **对照生成规则补齐** — 审核说缺什么，回看阶段2的规则补什么
3. **修复后必须重审** — 不能自己说修好了就过

### 修复流程

```
1. 读取审核报告，提取 P0/P1 问题列表
2. 对每个问题：
   a. 定位问题位置（哪个章节？哪个Task？）
   b. 回看阶段2对应规则
   c. 按规则补齐/修正
3. 生成 TODOLIST_v{N+1}.md（或直接覆盖）
4. 返回阶段2.5重新审核
```

### 常见问题修复

| 问题 | 修复方法 |
|------|---------|
| 缺少章节 | 按阶段2.4的6个必需章节补齐 |
| Task缺少部分 | 按阶段2.3的6部分模板补齐 |
| 功能要求<3条 | 从PROJECT_PLAN维度5提取更多功能点 |
| 缺代码审核命令 | 根据技术栈添加（pylint/eslint/等） |
| 进度不是0% | 改为0%，所有checkbox改为`[ ]` |
| 有循环依赖 | 分析实际依赖，打破循环 |

### 重试限制

最多修复 3 次。3 次后仍 <90 分 → 停止，输出：
```
⚠️ 修复3次仍未通过，请检查 PROJECT_PLAN 是否完整，或手动修改 TODOLIST
```

## 执行

现在开始执行完整流程：

1. 搜索并读取需求文档（必需）
2. 搜索解决方案文档（可选，有则读取）
3. 分析代码库判断项目阶段
4. 生成 PROJECT_PLAN_v1.md
5. **输出审核报告** → 修复循环（直到≥90分，最多3次）
6. 生成 TODOLIST.md
7. **输出审核报告** → 修复循环（直到≥90分，最多3次）
8. 输出完成总结
