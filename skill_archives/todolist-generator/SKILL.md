# 生成开发任务清单

## 执行步骤

1. 读取 PROJECT_PLAN.md（需已通过审核，评分≥90）
2. 定位维度5"核心模块设计"章节
3. **逐个模块执行以下操作**：
   a. 找到模块的"接口"部分（```javascript 或 ```python 代码块）
   b. **复制该代码块**
   c. 生成 Task，**将代码块粘贴到"核心接口"部分**
4. 分析模块依赖关系，拓扑排序
5. 生成6个必需章节
6. 执行输出前强制检查
7. 输出 TODOLIST.md

---

## ⚠️ 完整 Task 示例（必须严格遵循）

以下是一个**正确格式**的 Task，注意"核心接口"部分是**必须存在**的：

```markdown
### Task 1: 数据层实现

**状态**: 未开始
**优先级**: P0
**依赖**: 无

**功能要求**:
1. 加载 CSV 文件并解析数据
2. 实现按名称搜索功能
3. 实现数据清洗和验证

**核心接口**:
（⬇️ 从 PROJECT_PLAN 维度5 直接复制）
```javascript
const csvHandler = {
  loadDatabase(filePath) { },
  searchByName(query) { },
  getById(id) { }
};
```

**实现位置**:
- 文件: `src/data/csv-handler.js`
- 测试: `tests/test-csv-handler.js`

**代码审核**:
```bash
eslint src/data/csv-handler.js
```

**测试验证**:
```bash
npm test -- tests/test-csv-handler.js
```

测试用例:
- test_loadDatabase_success
- test_loadDatabase_fileNotFound
- test_searchByName_found
- test_searchByName_notFound

**验收标准**:
- [ ] CSV 文件加载成功
- [ ] 搜索返回正确结果
- [ ] 所有测试通过

**完成后必须执行**:
1. 确认代码审核通过
2. 确认测试通过
3. 更新 TODOLIST 进度
4. 检查下一个 Task 的前置条件
```

**⚠️ 注意**：如果你的 Task 没有"**核心接口**:"后跟代码块，格式就是**错误的**！

---

## ⚠️ 核心接口强制规则

1. 每个 Task **必须**包含"**核心接口**:"部分
2. 接口代码块**必须从 PROJECT_PLAN.md 维度5复制**，不能自编
3. 格式必须是：`**核心接口**:` 后面跟代码块
4. **缺少核心接口 = Task 不完整 = 禁止输出**

**具体操作步骤**：
1. 打开 PROJECT_PLAN.md
2. 找到维度5"核心模块设计"
3. 找到每个模块的接口代码块（```python 或 ```javascript）
4. **原样复制**该代码块
5. 粘贴到 Task 的"核心接口"部分

---

## 依赖分析

从以下位置提取依赖关系：
- 维度5：模块的"依赖"说明
- 维度3：数据流图
- 维度6：API调用关系

**拓扑排序规则**：
- 无依赖的模块排最前
- 有依赖的模块在依赖项之后
- 检测到循环依赖 → 报错停止

---

## Task 必需部分

每个 Task **必须**包含以下7部分：

1. **功能要求**（≥3条）
2. **核心接口**（从维度5复制的代码块）
3. **实现位置**（文件路径）
4. **代码审核**（具体命令）
5. **测试验证**（≥3个用例）
6. **验收标准**
7. **完成后必须执行**

---

## ⚠️ 输出前强制检查

生成完所有 Task 后，**必须执行以下检查**：

```
遍历每个 Task，检查是否有 **核心接口**: 后跟代码块：

□ Task 1 有核心接口代码块？
□ Task 2 有核心接口代码块？
□ Task 3 有核心接口代码块？
□ ...
□ Task N 有核心接口代码块？

⚠️ 只要有一个 Task 没有核心接口代码块 → 回去补充，不能输出
```

**检查方法**：在每个 Task 中搜索 `**核心接口**:` 后面是否有 ``` 代码块

---

## 每个 Task 生成后检查

生成每个 Task 后，立即验证：

```
□ 有"功能要求"且 ≥ 3 条？
□ 有"核心接口"代码块？（从 PROJECT_PLAN 维度5复制）
□ 有"实现位置"且有文件路径？
□ 有"代码审核"且有具体命令？
□ 有"测试验证"且有 ≥ 3 个用例？
□ 有"验收标准"？
□ 有"完成后必须执行"？
```

**任何一项缺失 → 补充后再继续下一个 Task**

---

## 必需章节

TODOLIST 必须包含6个章节（SPEC.md T-S1~S6）：

1. **进度追踪** - 显示总体进度和每个Task状态
2. **任务依赖关系** - ASCII图展示依赖
3. **任务详情** - 所有Task的完整内容
4. **首次执行指令** - 创建配置文件的指令
5. **工具说明** - 测试和审核工具说明
6. **注意事项** - 执行顺序和完成标准提醒

---

## 初始状态规则

新生成的 TODOLIST 必须满足（SPEC.md T-I1~I4）：

1. 进度 = 0%（不是100%）
2. 所有 Task 状态 = "未开始"（不是"已完成"）
3. 进度条显示 0%
4. 所有 checkbox 未勾选 `[ ]`（不是 `[x]`）

---

## 输出自检

生成后，对照 SPEC.md 4.2 检查：

```
□ 6个必需章节都存在？
□ 每个Task的7个部分都有内容？
□ 每个Task有"核心接口"代码块？
□ 每个Task测试用例 ≥ 3 个？
□ 进度 = 0%？
□ 所有Task状态 = "未开始"？
□ 依赖关系正确（无循环）？
```

**任何一项未通过 → 修正后再输出**
