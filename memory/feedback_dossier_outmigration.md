**通用文档组织纪律：总览表（清单/日历/速览表）里的条目，一旦从「一句话」长成「需持续追踪进展 + 判断会演进」的活体对象，就外迁成独立档案文件，总览表那格只留一句话指针（状态 + 链接）。**

**判据**：某条目还是"一行就说清的日历项/状态项" → 留在总览表；一旦它开始有多次进展、需要事实/判断分层、判断随时间演进 → 它"够格独立"，外迁。

**外迁后的结构**：
- 总览表那格 = 一句话（当前状态 + `见 XXX.md`），与表里其他一句话条目保持整齐。
- 独立档案 = 头部4要素（是什么/为什么盯/出手判据或下一步动作/怎么更新）+ 事实层（只增不改）+ 判断层（标日期，会变就改本区）。头部4要素见 [[feedback_docs_for_ai]]，分层见 [[feedback_tracking_facts_only]]。

**Why:** 2026-06-24 P9 趋势观察池 E1（FERC 事件）确立。错误示范：先把多段分层硬塞进总览表的一个格子，导致 E1 撑成五六行、而同表其他行都一句话，破坏总览表"一眼看全"的本意。主公点出关键新信息——"这类记录以后只会越来越多、一天多一条"，所以重点不是为单条低频事件费功夫，而是趁记录还少把"怎么归档"的范式立对，否则全堆总览表迟早撑爆、返工更贵。参考范本：legal_library Organic Blooms 诉讼追踪（独立 .md + 源表留指针）。

**How to apply:** 写总览类文件（事件日历/盯防清单/案件列表/项目台账）时，每加/改一条先判断它是"一句话项"还是"活体追踪对象"；后者直接外迁独立档案、总览留指针，别等内容堆混了再拆。新建独立档案默认就拉头部4要素 + 事实/判断两段骨架。与 [[feedback_self_organize_artifacts]] 同族（都是"产出长大了就给它专门的家 + 留索引指针"）。
