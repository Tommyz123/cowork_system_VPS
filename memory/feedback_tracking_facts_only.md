**通用规则（适用所有文件记录，非仅案件追踪）：数据与主观判断必须分层隔离。**

- **数据区 = 纯客观事实**，永远不掺主观判断/推测/赢面/动机分析，保持干净。
- **判断不是不能写，而是必须「分层」**：单独开一个区（如「分析层（YYYY-MM-DD）」），标注时点。
- **数据区零主观，判断区可留痕但隔离 + 标时点 + 带可验证条件。**

**两个执行要点（2026-06-13 加强，否则易做成形式）：**
1. **警惕"伪装成事实的因果判断"**：有些东西看着像数据其实藏了判断。如"December queue 被冻结"=事实，但"主要因为这个诉讼"=因果判断（队列慢也可能是 OCM 人手/流程）。连伪装成事实的因果推断也要拎到分析层，不留在数据区。
2. **判断要可被验证**：分析层不只标日期，还要标"基于哪几条数据 + 这个判断将来怎样算被证实/证伪"。否则未来回看只知"当时这么猜"，不知"猜对没、为什么"。

**Why:** 主公 2026-06-13（Organic Blooms 案 904497-24）确立。数据会增加、会变，方向和判断会跟着变。把"当时的判断"和数据混在一起，未来数据一变旧判断就成污染，且分不清哪句是事实、哪句是当时的猜。分层后能对比"当时是这些数据 → 当时这么判断"，判断对错可追溯，数据本身永远干净。

**How to apply:** 写任何长期沉淀的文件（案件追踪/参考库/进度/复盘等）时：数据区只录事实（原文、文件编号、日期、官方数字、当事人状态、时间线、摘录原句），并剔除伪装成事实的因果句；判断/推测/概率/动机另起「分析层（YYYY-MM-DD）」分区，每条判断带"依据数据 + 验证条件"。口头对话给主公的分析照旧标"推测+依据"。与 [[feedback_prediction_data_first]]、[[feedback_proposal_data_first]] 同源——事实与判断必须分层。

**写入即分层（2026-06-24 强化，主公明确要求"每次写入都要自动分层好"）：** 分层是**写入动作的前置自检步骤**，不是事后补救、不靠主公提醒。每次往沉淀类文件写记录前先自问"这句是事实还是判断？该进哪个区？"——混写就是写入未完成。反例：2026-06-24 P9 趋势观察池 E1 行，事实(show cause/复述报道)与判断(喜忧参半/门未开/出手时点)混在一格，靠主公点出才重整。**新建沉淀类文件/区块时，默认就拉出「事实」「判断·标日期」两段骨架**，别等内容堆混了再拆。
