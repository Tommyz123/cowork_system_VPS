⭐**文档读者二分总纲**（2026-06-23 主公确立）：

**判据一刀两断**："这东西是**发出来的**，还是**存着的**？"
- 📤 **定时发出去给主公的**（周报 / 邮件 / Discord 推送 / 客户材料）→ **给人看**：写得主公易读舒服（沿用 [[feedback_discord_long_doc]] 分段贴、[[feedback_preview_before_execute]] 邮件先预览）
- 💾 **留在系统里的**（playbook / memory / 脚本说明 / 索引 / 档案 / 作战表 / CLAUDE.md）→ **给 AI 看得懂**：第一读者是"未来那个没上下文的 AI"

**系统内部文档的合格标准**：一个新接手的 AI 读了它，能不能搞懂**整个系统的结构**——哪个文件管哪件事、各部分怎么分工、彼此什么关系，**不会混乱、不会误判**？能=合格；只有人能脑补=不合格。

**合格标准升级（2026-06-24 P9 E1 案，主公连环追问"AI 看得懂吗"逼出）**：从"看懂结构"再进一步到"**换一个全新 AI 能否独立接手继续这件事**"——不只可读，还要让无上下文 AI 知道"**下一步该干嘛**"。光记"发生了啥+当时怎么判断"不够，必须写出"什么情况下该采取什么动作"。**追踪/盯防类档案头部必写 4 要素：是什么 / 为什么盯 / 出手判据(或下一步动作) / 怎么更新**——其中"出手判据/下一步动作"是从"看懂历史"跨到"能接手"的关键那一步。

**Why**：主公遥控多 AI 实例（AA/BB/CC）+ 长期跨对话，文档真正的服务对象是"接手时没有上下文的 AI"。文档让 AI 看懂结构 = 系统能被正确维护、不被误改的根本。

**How to apply（这是总纲，下列规矩都是它的分身）**：
- 脚本/哨兵/自动化头部写四段：**这是什么 / 为什么做 / 为什么这样设计 / 演进史**（2026-06-23 FERC 哨兵案落地）
- 建文件必登索引，让 AI 知道"有这个文件、它管啥"（[[feedback_artifact_indexing]]）
- 文档间互相指针引用，让 AI 顺藤摸瓜（[[feedback_self_organize_artifacts]]）
- 数据与判断分层隔离，AI 不会把旧判断当事实滚雪球（[[feedback_tracking_facts_only]]）
- 改动在修订记录留"为什么改"，不只记"改了啥"
- 写系统内文档时随时自检：换个没上下文的 AI 来读，它会懂还是会糊涂？
