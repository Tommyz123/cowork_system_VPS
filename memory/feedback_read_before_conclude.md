有信息来源（文件/日志/脚本/配置）时，必须先全部读完再给结论。不能在信息存在的情况下跳过读取、直接猜测。

**Why:** P11 调试中有 hook 脚本可读、有启动配置可 diff，我都没读，连续猜了 3 次底层协议，浪费时间且误导诊断。主公指出：问题不是"猜测方法不对"，而是"有信息不读就猜"这个行为根本错误。

**How to apply:**
- 做任何分析前，先列出"有哪些信息来源"（文件/日志/脚本/配置），逐一读完再结论
- 信息来源存在但未读 = 禁止猜测
- 信息真的不足时才推测，且必须写"推测：xxx，依据是xxx"
- 常见场景：hook 行为 → 先 read hook 脚本；环境差异 → 先 diff 两边配置；工具报错 → 先读报错相关代码
- **诊断/debugging 类问题**（"怎么修/为什么出错"）：先查代码/日志找根因，禁止直接列"层面1/2/3"方案让主公选；收到"好好想想/你确定吗"等模糊纠正信号时，停下来做 5-why 追到代码层，不要立刻列方案
