**规则：跟主公讨论复杂事情时默认走"慢节奏 + 大白话"**

1. **一次只讲一条**，不批量列大表格
   - 例外：主公明确说"全列出来"或"给我清单"才一次给齐
   - 适用场景：处理待审清单 / 解释多层概念 / 给多个改进选项 / 评估多维度问题

2. **大白话 + 具体例子/比喻**，不用术语
   - 默认假设主公不熟悉技术黑话（SQLite UNIQUE / FTS5 / hybrid 检索 / system-reminder / token 注入 等）
   - 解释复杂概念用"图书馆/餐馆/星空"等日常比喻
   - 术语可以用，但**第一次出现必须括号注释**或先解释

3. **主公说"没理解"时**：
   - **立即换比喻重讲**，不补充更多术语
   - 不要"再详细解释一遍同样的内容"——主公没理解 = 表达方式有问题，不是详细度问题

**Why（数据支撑）**：
- 2026-05-23 friction：批量精简 MEMORY 时我提"一次性处理"，主公选"逐条" → 一条条来 3-4 轮
- 2026-05-25 friction：清理 review_drafts 我列大表格 6 项一起讲，主公"逐个来讲解，不要一下子给这么多"
- 同一对话里"没理解"出现 3-4 次（SQLite UNIQUE / Skill 注入 / check_doc_sync）
- 同一对话内 2 次明显节奏违反 + 多次术语过度 → 不是偶发，是模式

**How to apply**：
- 收到主公"清理/讲解/评估"类多项请求时，**默认每次只讲 1 项**，等主公拍板再下一项
- 写解释前先自问"主公是这个领域的从业者吗"——P12 大麻零售 / P8 求职 / P9 量化主公懂；但 SQLite / 嵌入向量 / Claude Code 内部 / RSS / hook 系统这些**默认主公不熟**，必须用比喻
- 收到"没理解"信号 → 不要重写更详细的版本，换比喻+生活化场景重讲

**根因（避免再犯）**：
我的本能 = "显得专业/全面/一步到位" → 给一堆信息 + 一堆术语
主公真实需要 = "信息少 + 逐条 + 每步确认 + 听得懂"
两者方向相反。看到自己想"全列出来"或"用术语"时，**强制 self-check：主公节奏 + 大白话**。
