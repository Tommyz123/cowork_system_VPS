**规则：听到"什么都行 / 都可以 / 看你 / 随便"等宽松词时，必须先追问 hard requirement 再开始讨论**

**追问标准句**：
> 「您是真的什么都行吗？有没有什么是绝对不接受的 / 必须满足的？」

也可以更具体（视场景）：
- 求职场景："薪资底线？地点要求？必须能学到的技能？"
- 选址场景："房租上限？最低人流量？必须避开的区域？"
- 选品场景："必须不能卖的品类？利润率底线？"

**Why（数据支撑）**：
- 2026-05-24 P8 求职讨论：主公开头说"什么都行 / 只要进 AI 公司"，我以为是宽松偏好，给了一堆公司选项
- 多轮后主公补充"必须能学到 + 用得上 cowork 经验" → 这是 hard requirement（硬约束），不是偏好
- 前面给的选项里有些违反这个 hard requirement，回头重新筛
- 损失：1.5h 讨论部分浪费

**根因**：
- 用户说"什么都行"经常**不是字面意思**——是当时没想到要先说"不接受 X"
- 类比：朋友点菜说"什么都行"，你点海鲜，朋友说"我对海鲜过敏" → 朋友不是字面"什么都行"，只是当时没想到要先告诉你过敏

**How to apply**：
1. 听到"什么都行 / 都可以 / 看你 / 随便"等词 → **立刻追问一句**，不要直接开始给选项
2. 追问时给具体维度（如"薪资？地点？技能？"），帮用户自己想清楚
3. 拿到 hard requirement 后再开始讨论选项
4. 如果用户后来又补"必须 X"，这是 hard requirement 不是 nice-to-have，必须立即用它筛选已给的选项

**反模式（避免）**：
- ❌ 听到"什么都行"就当成宽松偏好，开始堆方案
- ❌ 把后补的"必须 X"当成"主公变卦"，应该理解为"主公本来就有这个底线，只是现在才说出来"

**适用场景**：所有需求收集场景，跨项目跨主题通用。不只 P8 求职。
