主公明确反对全量加载所有记忆文件，认为是 token 浪费。
主公同样反对滥写记忆——乱写会让数据质量变差，记忆失去价值。

**核心需求：** 按需检索 + 严格写入标准。

**Why:** 记忆的价值在于"有用时能准确调用"，乱写导致噪音增加，反而干扰判断。

**How to apply — 读取：**
- 不要在会话开始就读取所有 memory 文件
- MEMORY.md 的摘要足够做第一层判断，需要深度细节时再读具体文件
- 未来 memory 文件超30个时可评估 mini RAG 路由

**How to apply — 写入（两层过滤）：**

**第一层：快速拒绝（遇到这些直接找对应位置）**
- 是待实现想法 → BACKLOG
- 是当前进度/状态 → CURRENT_SESSION.md
- 代码/文件里能读到 → 不重复
- 模糊愿望（"以后改进X"）→ 太虚，不写

**第二层：5个问题（越多Yes越应该写，少于3个不写）**
① **持久性** — 1个月后还准确吗？（状态会变的不写）
② **跨对话价值** — 新开对话的 Claude 不知道会吃亏吗？
③ **改变行为** — Claude 知道后会做不同的事吗？
④ **不可推导** — 无法通过读现有文件推出来吗？
⑤ **够具体** — 具体到能指导行为，而不是泛泛方向？

**不同类型的侧重：**
- feedback → 重点看③（改变行为），最高标准
- project → 重点看②④（跨对话价值 + 不可推导）
- reference → 重点看②⑤（跨对话 + 具体路径）
- user → 重点看①③（持久 + 影响沟通方式）

**How to apply — 决策变更（2026-06-06 主公确认）：**
当主公改变一个已记录的决定（今天定A，后来改B），不是删旧写新，而是「**更新最新状态 + 保留旧决定和变更原因**」。
- 同一问题的最新答案 → 修改那一条（像改通讯录里同一个人的电话号），永远只保留一个当前有效答案，禁止新旧两条并存（并存才会乱）
- 变更脉络 → 在该条后备注一句「原计划A，因[X]改为B」，方便回溯但不影响"当前用B"
- **真正风险不是覆盖、是"改了忘改记忆"** → 主公一旦改主意，当场更新对应记忆，不等提醒（这是我自觉该做的活）
