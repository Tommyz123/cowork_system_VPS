**Codex协作分工（2026-04-20主公确认）：**

- **我（Claude）**：理解主公意图 → 拆解任务 → 写清楚任务单 → 派给Codex → 验收结果 → 汇报
- **Codex**：执行层，负责写代码、跑脚本、处理文件等有明确输入输出的任务
- **不派给Codex的**：模糊任务、需要理解上下文的判断、与主公的对话

**后台进程规则：**
用 run_in_background 启动的命令，结果拿到后必须确认进程已退出，不留残留。

**Why:** 今晚测试时until循环跑了11分钟未退出被主公发现，同时确认了执行类任务应由Codex负责的分工模式。

**How to apply:** 收到执行类任务时，先设计好任务单再派Codex，不要自己直接跑Bash；后台命令完成后检查进程状态。

**Codex/eval 后台执行规则（2026-04-20主公要求）：**
派 Codex 执行耗时任务（写代码/跑测试）时，必须使用 `run_in_background: true`，不等结果阻塞主线程，跑完自动通知，期间可继续处理主公其他事情。eval 后台跑同理。

**Codex指令写法规则（2026-04-24主公确认）：**
①指令控制在<800字，只写"在哪+做什么+核心约束"；②背景/历史/设计决策不粘贴进指令，给文件路径让Codex自己读；③Codex有Bash工具可读本地文件，路径必须在prompt里明确指定。

**Why:** 长对话+长Codex指令双重消耗每日token配额，之前几千字指令是主要原因。

**WSL路径限制（2026-04-20）：**
/mnt/c/... 是 Windows 盘的 WSL 挂载路径，Codex 沙盒无写入权限。
→ 派 Codex 处理此类文件时，prompt 加："文件在WSL挂载盘，你无法直接写入，只需提供完整diff/patch，我来应用"

**Codex改动后文档标记（2026-05-07主公确认）：**
Codex 执行任何文件改动后，Claude 必须同步在 cowork_log.md 写带 `[需同步: ...]` 的条目。
**Why:** P9升级时Codex改了脚本但Claude没写[需同步]标记，导致ARCHITECTURE.md/context.md描述旧系统长达数周。
**How to apply:** 任何项目，Codex汇报完成后，Claude立即评估哪些系统文档需要同步并写入日志条目。

**子agent派出判断标准（2026-05-08主公确认）：**
一句话判断法："任务完成后我还需要记住过程细节吗？"
- 需要 → 自己做
- 不需要 → 派出去

具体对应：
- 改代码/验证 → Codex
- 大量工具调用的调研 → Fork agent
- 搜索/读文件/即时操作 → 自己做

**Why:** 解决了之前派Codex时机模糊的问题，一句话可判断而不需要列清单对照。
