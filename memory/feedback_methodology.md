**主公核心方法论（4步）：**

1. **最小可验证一步起步** — 不从完整方案开始，找最小的可验证单元
2. **同步建监测机制** — 启动的同时定好"怎么知道它工作了/出问题了"
3. **数据/事件触发才升级** — 不靠预感，靠 friction_log / 指标 / 事件决定下一步
4. **已研究方案文档留底等触发** — 研究过的方案写成文档留着，不是现在不做，是等触发条件

**与 feedback_yagni 的区别：**
- YAGNI = 没需求就不做（静态）
- 这条 = 渐进做 + 留底（动态）。允许"研究但先不做"，等触发再捡起来

**Why:** 主公原话 2026-05-19 21:21 EDT："我们从小做起，就是先做一个收益率最高的，然后设置提醒，如果还有那个问题，就做哪个，保留技术，然后给项目升级做后备"

**How to apply:**
- 主公问"要不要做 X？"时，先帮他拆出 X 的最小起步版 + 触发条件
- 不要直接推全功能方案
- 已研究但暂不做的方案 → 写进 BACKLOG（含触发条件），不要丢
