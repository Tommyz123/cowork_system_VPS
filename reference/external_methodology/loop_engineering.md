## 一、核心思想（一句话）

- **Prompt Engineering（旧）**：人坐电脑前**一句一句喂** AI——发指令→看结果→不满意再发→再看…全程靠人盯。
- **Loop Engineering（新）**：人**设计一套系统去驱动** AI——AI 自己**触发→干活→验证对不对→不对就重试→搞定才停并通知人**。人从「打字员/司机」变成「设计循环的工程师/造自动驾驶的人」。

**关键认知转变**：「价值单位」从**单次回答（response）**变成**整条轨迹（trajectory）**。AI 第一次出 bug 不要紧，只要系统能自测出 bug → 跑测试 → 第四次修对了，就算成功。

**一句精辟总结（Sonar 提出）**：**"没有验证的 loop engineering，只是普通自动化（automation）。"** —— 验证才是 loop 和死板自动化的分水岭。

## 二、一个 loop 的 7 个零件（设计循环时逐个对照）

| # | 零件 | 说明 | 设计时问自己 |
|---|------|------|------|
| 1 | **触发器 Trigger** | 什么时候启动（schedule / 事件 / 用户命令）| 这个 loop 什么时候该跑？ |
| 2 | **目标 Goal** | 具体、**可验证**的目标（不是"改进代码"这种空话）| 怎样算成功？能被检验吗？ |
| 3 | **上下文 Context** | 给 AI 的资料和规则 | 它需要知道什么才能干好？ |
| 4 | **动作 Action** | 允许用的工具和能力 | 它能调哪些工具？边界在哪？ |
| 5 | **验证 Verification** | **确定性检查**（跑测试 / linter / CI 状态）——**不是让 AI 自己说"我觉得对了"** | 用什么客观标准判对错？ |
| 6 | **状态 State** | 结果记在哪，供下一轮迭代用 | 这轮结果存哪？下轮怎么读到？ |
| 7 | **停止/上报 Stop/Escalate** | 退出条件 + 何时交回给人 | 什么时候停？什么时候叫人？ |

> ⚠️ **第5零件「验证」是灵魂**：loop 能不能自我纠错，全看验证是不是「确定性的客观检查」。让模型自己打分 = 假验证（它会自我感觉良好）。

## 三、4 种循环类型（按触发方式分，含成本与失败模式）

| 类型 | 触发方式 | 例子 | 对应本系统 | 月成本(Sonnet 4.6) | 典型失败模式 |
|------|---------|------|-----------|---------|---------|
| **心跳 Heartbeat** | 每隔几分钟一直转 | 每10min查工单队列 | instance_watchdog（每5min查三实例）| ~$13（500tok/5min）| 并发重叠处理→需幂等锁 |
| **定时 Cron** | 固定时间跑 | 每周一9点 | P9 全套 cron / 每日新闻 | ~$0.06（5000tok/周）| **陈旧指令**：几月前调的prompt一直跑没人发现退化 |
| **事件 Hook** | 被事件触发 | PR提交/CI失败/告警 | hook体系(UserPromptSubmit/Stop)| ~$60（2000tok/工单×1000）| **webhook风暴**：上千事件→上千次跑，几小时烧光预算→需限流 |
| **目标 Goal** | 反复迭代直到达成才停 | 改代码直到测试全过 | 当前系统较少，最"智能"的一类 | ~$9（60000tok×5次/月）| **最贵**：目标无收敛条件→永不停止烧token |

> 💡 **成本洞察**：心跳=固定成本（按频率定），目标=可变成本（按难度定）。**多数生产级 agent 是混搭多种类型**，不是只选一种。

## 四、和 Prompt Engineering 的区别（一表）

| 维度 | Prompt Engineering | Loop Engineering |
|------|-------------------|------------------|
| 作用范围 | 单次交互 | 重复的整个流程 |
| 假设 | 第一次回答就能用 | 输出需要验证 |
| 人的角色 | 坐在椅子上手动跟进 | 设计系统，系统决定下一步 |
| 验证 | 人工follow-up | 自动化确定性检查 |
| 优化对象 | 提示词本身 | 提示词周围的自主行为系统 |

## 五、5 大反模式（Anti-Patterns）+ 4 道强制护栏

### 5 大反模式（踩了就翻车）
1. **失控成本（Runaway costs）** —— 所有类型的头号杀手
2. **重叠执行** —— 心跳无幂等→同一活干两遍
3. **陈旧指令** —— cron 跑着几月前的老prompt，悄悄退化
4. **无限扩张** —— hook 无限流→事件风暴烧预算
5. **无限迭代 / 过早完成** —— 目标无停止条件→永远跑；或半成品谎报"已完成"还继续烧钱（premature-completion，最隐蔽）

### 4 道强制护栏（生产级 loop 缺一不可）
1. **每loop迭代上限**（如最多50轮）
2. **每日预算上限**（token花费封顶）
3. **可评估的成功条件**（"简报完成"❌ → "覆盖竞品X/Y/Z且有具名来源"✅）
4. **上报路径**（目标达不成→通知人，不能默默继续）

> 🔑 **最核心的设计问题（业界共识）**："**谁有资格说这个 loop 干完了？**" —— 如果判断"完成"的是**干活的同一个模型**、或**另一个被叫来review的模型**，那是"两个乐观主义者互相点头"。**理想的验证应来自确定性检查（跑测试/外部事实），不是模型自评。**

## 六、怎么套用到本系统（防过度工程红线）

> 主公定的要求（2026-06-23）：**套用要"刚好用得上"，不为了用而用，不过度工程。**

套用任何系统前，先过这 3 道关（呼应 [[feedback_yagni]]）：

1. **有没有真实的反复痛点？** —— 没有人反复手动做的事，就不需要 loop。（信号天天人工看才需要自动榜；一次性任务不需要）
2. **验证能不能自动化？** —— 第5零件是灵魂。若"对错"只能靠人主观判断、没法写成确定性检查，那这 loop 做出来也是假闭环，不如保留人工。
3. **失败代价可控吗？** —— 涉及下单/不可逆动作的 loop，停止/上报零件必须先设计好（P9"thesis失效只告警不自动平仓"就是这条的体现）。

**判定公式**：真实反复痛点 ✅ + 验证可自动化 ✅ + 失败代价可控 ✅ → 才值得做成 loop；缺一 → 保留人工或只做半自动。

## 七、本系统现状盘点（用 loop engineering 视角看 cowork/P9）

主公早就在做 loop engineering，只是没这个名字：

| 已是 loop 的 | 类型 | 7零件齐了吗 |
|---|---|---|
| P9 cron 全套（采信号/对账/告警/周报）| 定时 Cron | ✅ 基本齐，验证=对账脚本（确定性✅）|
| instance_watchdog | 心跳 Heartbeat | ✅ 齐，注意"只通知不重启"=人工上报路径 |
| hook 体系（授权/日志/诚信检查）| 事件 Hook | ✅ 齐 |

**缺口（验证→状态反哺那环没接上的"半loop"）**：
- P9 `theme_discovery`：信号天天采（触发✅）但没有"从信号自动发现主题→验证→反哺选股"的闭环——缺第5/6零件。
- ⚠️ 本系统多数 cron 是"定时"类，要警惕反模式#3「陈旧指令」：几月前调的 prompt 一直跑，要定期审视是否退化。

> 详细套用方案 = 下次需要时，对照「7零件 + 5的3道关 + 4道护栏」逐项填，就知道怎么搭、且不会过度工程。

## 八、什么时候调出本文件

- 主公说"给 XX 加自动循环 / 自动化 / loop / 让它自己跑 / loop engineering"
- 设计任何 cron / hook / 自动迭代机制前 → 用 7 零件对照设计 + 4 道护栏检查
- 评估"这个自动化值不值得做" → 用第六节 3 道关筛
- 审视现有系统"哪些是真闭环、哪些是半 loop / 哪些 prompt 陈旧了" → 用第七节视角
- 排查自动化"烧钱/跑飞/谎报完成" → 用第五节 5 反模式对照
