## 二、MiroFish 预测原理 = 六步涌现流水线（已读代码确认）

> 关键修正：原以为五步，主公揪出漏了"环境"，改为六步。

```
① 抽实体(ontology)
   ↓ 只抽"现实中真实能公开发声/互动的主体"，禁止抽象概念
② 建知识图谱(Zep GraphRAG)
   ↓ 把实体+事实+关系连成图，图就是信息差和可见性的载体
③ 长角色(persona)
   ↓ 每个实体从 Zep 取回它自己的真实事实，长成~2000字人设
③.5 造环境(OASIS 起 Twitter/Reddit 舞台)  ← 之前漏的关键一步
   ↓ 有规则的互动场：可见性规则+动作集+回合时钟
④ 角色丢进环境多轮碰撞
   ↓ 每轮激活一批agent，各自从动作集选动作；发言实时回写图谱
⑤ ReACT 报告agent 提取预测
   ↓ 上帝视角，每章节必调3-5次检索工具、禁用自身知识、必引agent原话；不出概率数字
```

### 各步代码出处（本会话已读）
| 步 | 文件 | 关键点 |
|----|------|--------|
| ① | simulation_config_generator.py | `_generate_event_config` 生成 initial_posts，每条带 poster_type |
| ①信息差种子 | 同上 `_assign_initial_post_agents` | 把每条帖按 poster_type 派给身份匹配的具体 agent（官方声明→Official，新闻→媒体）= 信息差的物理种子 |
| ② | zep_entity_reader.py | `EntityNode` 带 related_edges(方向in/out+fact) + related_nodes，图→角色的桥 |
| ③.5+④ | run_parallel_simulation.py | `actions={agent:LLMAction()}` → `env.step(actions)`；Twitter 6动作/Reddit 13动作 |
| ④回写 | zep_graph_memory_updater.py | `AgentActivity.to_episode_text` 把每个动作转自然语言实时写回图谱→涌现更深 |
| ⑤ | report_agent.py | 上帝视角ReACT，4工具(insight_forge/panorama_search/quick_search/interview_agents)，必引原话，不算概率 |
| ⑤访谈 | zep_tools.py:1272 | `interview_agents` 重新唤起活着的OASIS agent第一人称作答，跨双平台合并 |

### OASIS 是什么（环境引擎真身）
- 全称 Open Agent Social Interaction Simulations，开源大规模社交模拟器
- 在 MiroFish 里 = "环境"本身，提供三样：①可见性规则(谁关注谁→谁看到谁) ②动作集(Twitter6/Reddit13) ③回合时钟
- Twitter/Reddit 只是两种舞台规则，**不是预测目标**；跑两个为交叉验证
- 没有这个场，角色各说各话，碰撞无从发生

## 三、拆坑记录（我犯过的坑，铁律来源，必须固化）

> 格式：每个坑 = 错在哪 + 根因 + 以后怎么避。前5个是**预测方法的坑**，后4个是**做事方式的坑**。

### A. 预测方法的坑（直接导致 v2 预测儿戏）

**坑1 假信息差（最致命）**
- 错在哪：同一批事实喂给所有角色，只给每个角色贴个"立场标签"，让它们碰撞 = 假碰撞，结论早就被我预设了。
- 根因：没理解 MiroFish 信息差是**物理造出来的**——靠 poster_type 把不同初始帖派给不同身份的 agent + 图谱关注关系决定谁看到谁。
- 怎么避：每个角色只拿它身份**该接触**的信息子集（见 fact_ledger 的 E 角色信息分发表），明确写出它的盲区。

**坑2 种子事实没核实就开跑**
- 错在哪：和解破裂误判日期、冻结范围搞反（真相是"只冻 no secured location 那批"，非整个 Dec 队列）。
- 根因：急着出结果，事实层 🟡/⚠️ 一堆没核实就当 ✅ 用。**种子不准则涌现全错**。
- 怎么避：每条事实标确信度(✅/🟡/⚠️/❓)，关键变量(如 D1 主案程序状态)未知就**在预测里明确标"未知"**，不脑补。

**坑3 自创概率数字**
- 错在哪：拍脑袋甩出"40%""50-55%"。
- 根因：以为预测就得给个数。但真版报告 agent **不输出概率**，只调检索工具+引 agent 原话推演走向。
- 怎么避：不报百分比，报"哪些力量推向哪、依据是谁的什么话"。

**坑4 没学会 MiroFish 怎么找信息源/角色扮演**（主公明确点的）
- 错在哪：跳过了"怎么找信息源、怎么造信息差、怎么让角色各不相同"就开跑。
- 根因：只看了表面流程，没读进去机制。
- 怎么避：先吃透六步流水线每一步的代码实现，再动手。

**坑5 漏掉"环境"这一环**
- 错在哪：把环境(OASIS)糊进"碰撞"步，五步框架里没单列。
- 根因：没意识到没有"有规则的互动场"碰撞根本无从发生。
- 怎么避：环境是独立的第③.5步，订阅版里我亲自当环境引擎。

### C. 引擎实现的坑（2026-06-08 补 MBTI 时暴出）

**坑10 parse_json 永远先试 `[...]` → 对象内嵌数组被误抓**
- 错在哪：加 MBTI 后所有角色 MBTI 全空(0/7)。
- 根因：`parse_json` 抽 JSON 死板先试 `[`。角色 JSON 是对象、里面嵌了 `interested_topics":[...]` 数组 → 函数抓到的是**内层那个话题数组**（它本身合法 JSON）、整个角色对象反而丢了 → 全走兜底默认值。
- 怎么避：改成**按 `{` 和 `[` 谁先出现当最外层**匹配（对象输出以 `{` 开头、纯数组输出以 `[` 开头，两种都对）。另 step3 加空解析重试1次硬化偶发超时噪声。教训：加字段后要**实跑验证字段真填上**，不能只看不崩就以为对了。

### B. 做事方式的坑（主公要求"写入和改正"）

**坑6 急着执行，没理解原理就跑**（主公最强调的一条）
- 错在哪：拿到任务立刻跑预测，被批"很儿戏"。
- 根因：把"动起来"当成"在干活"，跳过理解。
- 怎么避：**理解原理才执行，不是一开始就着急执行**。→ 这条要写进 CLAUDE.md（Task#6）。

**坑7 过早宣称"全部读透"**
- 错在哪：代码没读完就说"全读透了"，被主公连追"你确定？"
- 根因：用印象代替事实，违反数据诚信。
- 怎么避：读取覆盖度**分层老实交代**(已读/读了核心非逐行/没读)，不夸大。

**坑8 不读就判"无关紧要"**
- 错在哪：把 zep_tools 等判成低价值不读，被纠正"没读取怎么知道"，读后纠正2处误判+发现回写机制。
- 根因：凭文件名/印象下判断。
- 怎么避：判断价值前先读，至少扫核心。

**坑9 Discord 漏发 reply（贯穿全天，反复踩）**
- 错在哪：把回复写成 transcript 文本，没调 reply 工具，被 Stop hook 拦了 ~6 次。
- 根因：习惯性输出文本，忘了主公在 Discord 端看不到 transcript。
- 怎么避：所有给主公的回复**必须**走 `mcp__plugin_discord_discord__reply`，参数名是 `text` 不是 `content`（用 content 会报 text.length 错）。

## 五、今天探讨的"用三实例当大脑"方案（⚠️未验证，发现障碍）

主公想法：把 OASIS 的 API 调用改成用 AA/BB/CC 三实例（订阅版的我）来跑——改接入层不改 OASIS 主体。

### 可行性分析
- **注入点干净**：MiroFish 造大脑只一处 `ModelFactory.create`(run_parallel_simulation.py:1034)，所有agent共用一个model对象，换掉它即全换。
- **架构成立**：camel 模型都继承 `BaseModelBackend`，核心方法 `.run(messages)`。可写 `SubAgentModelBackend` 子类让 `.run()` 不发HTTP、改派子agent/实例生成。OASIS 不知背后换了人。

### 真实障碍（必须验证才能确认路通）
1. **方向相反**：OASIS 的 `.run()` 是它自己循环里**主动调**；子agent/实例是我**被动派**。要硬接得把 OASIS 改成可暂停/落盘的异步握手（暂停→外部填答→恢复），工程量大。
2. **三实例接入是 tmux send-keys + 抓屏**（敲键盘+读屏），不是函数调用：没返回值靠抓屏判断说完没、并发只3路、卡授权会挂。
3. **致命发现：三实例不是空闲资源**——抓 BB 屏幕发现它正 `Computing…` 跑自己的任务。AA/BB/CC 是**有主的活会话**，不是待命算力池。借用会打断它们工作、会话串台。
   → 结论修正：要用实例当大脑，须**专门另起干净的、没人用的临时实例**当工人，不能借用 AA/BB/CC。

### 下一步（待主公选）
- 写最小验证脚本，只测"send-keys 调用→稳定抓屏返回"这一个最小闭环。
- 推荐：另起一个临时 tmux 实例当靶子测，不碰 AA/BB/CC。

## 八、订阅版 vs 原版能力对照（2026-06-08 读源码后诚实评估）

> 进度已超越上面「六」的快照：引擎 `prediction_engine/predict.py` 六步全跑通，6角色/3轮含MBTI+图谱+碰撞+报告，大麻案实跑出完整预测报告。本节是读完原版源码后对"复刻了多少"的诚实打分。
> 读过的原版源码：simulation_runner.py(1768行)、zep_graph_memory_updater.py(554)、graph_builder.py(506)、report_agent.py(2572)、oasis_profile_generator.py、ontology_generator.py(506)、simulation_config_generator.py(991)。

**一句话结论：复刻了「方法论骨架」≈35-40% 工程能力，但拿到了 ~80% 的「原理价值」。**

| 维度 | 原版 | 订阅版 | 比率 |
|---|---|---|---|
| 六步流水线骨架 | 抽实体→图谱→角色→环境→碰撞→报告 | 同 | ✅~90% |
| 角色人设 | persona+MBTI+职业+话题+年龄性别国籍 | persona+MBTI+职业+话题 | ✅~75% |
| 知识图谱 | Zep GraphRAG，**每个动作回写**、长期记忆持续演化 | 一次性抽取，**碰撞不回写** | ⚠️~30% |
| 环境引擎 | 真 OASIS：Twitter/Reddit 多动作集 + **RecSys 推荐系统**决定可见性 | 自写简化版：post/reply + 知识子集做可见性 | ⚠️~35% |
| Agent 规模 | **成千上万** agent | **6-8 个** | ❌~5% |
| 上帝视角动态注入变量 | 跑中途可注入变量看分叉 | 没做 | ❌ 0% |
| 报告 agent | 2572 行 ReACT 深度推演 | 一次 sonnet 出报告 | ⚠️~40% |

**两个最大差距**：
1. **没有真·持续记忆图谱**（Zep GraphRAG 回写）——原版 agent 越聊图谱越丰富、记忆越长；订阅版图谱抽完即冻。
2. **规模差两三个数量级**——原版几千 agent 跑真"涌现"，订阅版 6 个只能算"小型圆桌推演"，跑不出群体涌现。

**为什么原理价值仍有 ~80%**：核心机密全复刻对了——信息差靠图谱物理造、角色扮演破法（标注合成数据绕开拒答）、引原话不报概率、六步顺序、换任何题目代码一行不改、零 API 费。

**定位**：当"小规模快速推演工具"完全够用；当"几千 agent 大规模涌现引擎"差得远。

**升级路线（要补真能力时按此走）**：① 接 Zep GraphRAG 让碰撞发言回写图谱（补最大差距1）→ ② 扩 agent 规模 + 引入 RecSys 式可见性（补差距2）→ ③ 加上帝视角动态注入变量。
