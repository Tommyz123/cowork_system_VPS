## 主线定位

**不是简单的"开一家大麻店"，而是**：
1. **第一层**：拿 NY 大麻零售牌照 + 自营一家店作为实验场
2. **第二层**：用 AI 把店内运营 / 顾客 / 营销系统化，做出差异化
3. **第三层**：把 AI 模块产品化 → SaaS 卖给 NY 100+ 家 dispensary
4. **终极**：店是分母（验证、数据源），SaaS 是分子（财富放大器）

**为什么是主线**：
- 命格层：辛卯大运（34-43 岁）偏财+偏印窗口，对应"创业财+新兴行业+智力变现"
- 行为层：主公的 AI 系统能力 + 法律积累 + 行业理解 = 三件套对齐
- 市场层：NY 大麻 + AI 是双拐点，时间窗口在 2026-2029（伤官生财年）

### ⭐ 创始人身份认知（底层定位，比技术细节重要 10 倍）

**主公的真实身份从第一天就要清楚**：
- ❌ 你不是"大麻店老板"
- ✅ 你是"**大麻零售 AI 工具的创始人，自营一家店作为种子用户和实验场**"

## ⭐ 架构核心原则：IP 边界 / 自建 vs Flowhub 边界

**这是这套系统的护城河逻辑，比任何技术细节都重要。**

### 核心比喻
**Flowhub 是你租来的基础设施，AI 系统是你自己的房子，地契永远在你手上。**

### 边界设计

```
┌─────────────────────────────────────┐
│  Flowhub（只做通用 POS，租来的基础设施）│
│  - 交易处理                          │
│  - 基础库存                          │
│  - 顾客基础信息（手机号、姓名）       │
│  - Click & Collect 下单 API          │
└──────────────┬──────────────────────┘
               │ webhook / API（单向数据流）
               ↓
┌─────────────────────────────────────┐
│  你自建系统（IP 重地，秘方所在）      │
│  - terp profile 数据库               │
│  - 6 维度产品特征工程                │
│  - 顾客行为分析（去 PII 后）          │
│  - 推荐模型（训练 + 推理）            │
│  - AI 推荐员对话引擎                 │
│  - 营销决策 AI                       │
│  - 数据中台                          │
└─────────────────────────────────────┘
```

### 关键边界原则

- Flowhub **只知道**：顾客 A 买了产品 X（必要的交易信息）
- Flowhub **不知道**：为什么推荐 X / 顾客 A 的 terp 偏好向量 / 推荐算法权重
- 这条边界 = 你的 IP 护城河

### 为什么必须自建

1. **平台方天然有数据访问权**——即使合同保密，Flowhub admin 看得到
2. **Flowhub 可能下场做**——他们看到你跑通了，第二年自己出个 AI 包
3. **IP 上别人平台 = 不是你的了**——SaaS 圈血泪教训

### 附加好处

| 好处 | 说明 |
|------|------|
| 数据可移植性 | 将来换 Dutchie / Treez，模型和数据不丢，只需要重写一层 adapter |
| SaaS 化通用性 | 卖给其他店时，他们用什么 POS 都能接 |
| 不被绑架 | Flowhub 涨价 / 改 API / 倒闭 → 你换就是了，AI 大脑在自己手里 |

## 技术栈选型

### 基座（用现成的，不自建）
- **POS 系统**：**Flowhub** ⭐ 已选（2026-05-14 决策）
  - 选择理由：API 开放性最好（三大 POS 里第一）、webhook 完整、第三方集成丰富、价格相对 Dutchie 便宜（适合单店起步）
  - 劣势接受：NY 客户基数比 Dutchie 小、社区资源少（不靠社区，自己做 AI 增强层）
  - 自带：库存追踪、合规报表（NY OCM）、基础 CRM、API 开放
  - **关键点**：用 Flowhub 作基座，不自己从零搭 POS，6-12 个月工程成本省下来投入 AI 层
- **支付**：大麻零售合规支付方案（CanPay / Hypur 类，待后续选型）

### AI 增强层（主公自建，核心差异化）
- **AI 推荐引擎**：基于 terp profile + 6 维度做个性化推荐
- **AI 推荐员**：QR code 入口的对话式推荐系统
- **库存预测**：基于销售数据预测进货量
- **营销自动化**：EDDM 后续追踪 + 复购触发 + 内容生成
- **合规审计辅助**：自动扫描交易日志找异常
- **客户特征系统**：手机号 + 偏好 + 购买路径

### 数据中台 ⭐ 自建（2026-05-14 决策）
- **顾客数据完全自建**：不依赖 Flowhub 内置 CRM 的标准化数据结构
- 选择理由：
  1. **训练自己的推荐模型**（模型权重和数据都在自己手里）
  2. **自己设计特征工程**（terp profile、效果标签、价格区间）
  3. **数据完全可控**（合规审计权 + 未来 SaaS 化时的核心资产）
- 架构方向：
  - **数据流水线**：Flowhub webhook → 自建数据库 → 特征工程层 → 模型训练 / 推理服务
  - **存储**：复用 Cowork 系统基础设施（cannabis_retail 独立 DB）
  - **隐私合规**：参照 HIPAA 类标准，opt-in 收集 + 加密存储 + 审计日志
- 工程成本认知：
  1. **数据流水线开发**：估计 2-4 周（首期）+ 持续维护
  2. **合规审计责任**：自建系统的 HIPAA 类合规责任全在自己
  3. **模型冷启动**：前 3-6 个月数据少，初期推荐质量低于规则引擎；需要"规则 + 模型"混合方案过渡

## ⭐ 推荐系统核心 IP：Terp-based Feature Engineering

**这是这套 AI 系统的差异化护城河，是真正的核心 IP。**

### 为什么是 Terp-based

- 学术界已证明：单纯 indica/sativa 分类已被现代大麻研究否定
- Terp profile + cannabinoid ratio 才是真正影响体感的变量
- Russo 等学者证明的 entourage effect（协同效应）
- Leafly 2025 年也承认了 indica/sativa 分类的局限

### Terp 数据基建（NY 监管帮你做了 80%）

- NY OCM **强制要求每批产品的 COA 必须包含 terp profile**
- 所有 NY 持牌产品 → 都有 terp 数据
- 数据基建免费，唯一要解决的是工程对接（从 testing lab / 持牌生产商 / Flowhub 进货数据抽 terp）

### 6 维度推荐模型

| 维度 | 来源 | 说明 |
|------|------|------|
| **1. 口味（Flavor）** | terp profile | 柑橘→limonene / 松香→pinene / 薰衣草→linalool / 胡椒→caryophyllene |
| **2. 感觉（Effect）** | terp + cannabinoid | 镇静→myrcene / 提神→limonene / 专注→pinene / 抗焦虑→linalool |
| **3. 价格（Price）** | 产品价格 + 客户客单价历史 | 作为筛选 + 推断价格敏感度 |
| **4. 使用形式** | 产品类型 | flower / vape / edibles / pre-rolls / tincture |
| **5. 使用时段** | 用户提供 + 历史推断 | 白天 vs 晚上（不同 terp 适合不同时段） |
| **6. 耐受度** | 历史 THC 含量分布 | 新手 vs 老手（影响 THC 强度推荐） |

### 推荐公式
```
推荐 = 匹配(口味偏好) × 匹配(效果场景) × 在(价格范围)
     × 物理形式匹配 × 时段匹配 × 耐受度匹配
     → 排序输出
```

### 为什么难被抄

- 需要懂 AI + 懂大麻化学 + 接得通 NY 数据流 三个交集
- 这个交集人才在市场上极少
- 别人抄你的 UI 容易，抄你的算法和特征系统 = 2-3 年追赶

### 合规边界

- AI 不能直接说"治焦虑 / 治疼痛"（FDA 红线）
- 要说"很多顾客觉得这款适合 social / unwind / focus 场景"
- 推荐内容必须有合规过滤层

## ⭐ 完整 AI 闭环架构（10 层）

```
[获客层]
EDDM 卡片投递 + 周边人口画像
    ↓
[入口层]
AI 推荐员 (QR) + 店内 QR（桌牌/小票/海报）+ budtender iPad + 自助 kiosk
    ↓
[转化层]
AI 对话推荐 → 加购物车 → Click & Collect 下单 → 到店取货 → ID 核验
    ↓
[数据层]
Flowhub 交易数据 + 自建顾客中台 + AI 自动打标签
    ↓
[运营层]
AI 库存预测 + Assistant Manager（员工管理）+ 持牌人合规签字
    ↓
[营销层]
AI 营销内容生成 → 人审 → 投递（EDDM / 邮件 / 短信，opt-in 后）
    ↓
[审计层]
合规 AI 报警 → 持牌人人工审 → 修正
    ↓
[迭代层]
A/B 测试 → 月人工抽检 50 单 → 模型升级
    ↓
[复购层]
复购触发新一轮 AI 推荐
    ↻ 回到入口层
```

### 5 个缺口的填法

**缺口 1：店内体验**
- 多点 QR 投放：桌牌 / 小票 / 产品展示牌 / 墙上海报
- budtender 配 iPad，实时查产品 + 推荐参考
- 店内自助 kiosk（2-3 个屏幕）— NY 大麻店现在基本没有，是真差异化机会

**缺口 2：下单方式**
- NY 现状：cannabis delivery 暂时禁止
- 主流：**Click & Collect**（线上下单 → 到店取货）— NY 允许
- Flowhub 自带 Click & Collect API，AI 推荐员直接对接

**缺口 3：客户分层**
- 3 档：新客 / 老客（≥3 单）/ VIP（前 10% 消费额）
- AI 自动打标签：偏好类型 / 用药场景 / 价格敏感度
- 策略：新客=首单折扣 / 老客=新品试用 / VIP=专属预约 + 限量产品

**缺口 4：人在 loop（合规审计边界）**
- 推荐 AI 全自动 OK（推错产品不致命）
- **营销内容 AI 生成必须人审才发**（邮件 / 短信 / EDDM 卡片）
- 合规审计 AI 报警 → 持牌人（你）人工审
- AI 出错的合规罚款 = 牌照风险，这层不能省

**缺口 5：模型迭代**
- 北极星指标：推荐转化率 / 客单价 / 复购率
- A/B 测试：每周拿一小部分用户跑新模型对比
- 每月人工抽检 50 单推荐质量
- 数据攒到 10K 单后切换"在线学习"模式

```
EDDM 寄出 → 周边居民收卡片 → 扫 QR
```

- **QR 入口**：卡片 / 包装 / 店内桌牌 / 小票

### 模块 2：店内 AI Budtender 辅助

### 模块 3：库存 AI 预测
- 基于销售数据预测进货量
- 减少过期 / 缺货损失

### 模块 4：AI 法律顾问 ⭐ V0 优先做（2026-05-14 升级）
**从"事后合规审计" → "实时法律审查"**
- 实时审 EDDM 卡片设计（21+ 警示 / 医疗用词）
- 实时审 AI 推荐内容（不能说医疗 claims）
- 实时审营销邮件 / 短信 / 员工培训材料
- 实时审库存合规（COA、产品来源、批次追踪）
- 实时审客户数据收集流程（opt-in 合规）
- 监管检查准备（自动生成 OCM 报表）
- 月度 / 季度 OCM 报表辅助生成（保留原合规审计功能）

**核心边界（必看）**：
- AI 可以提示"根据 NY 大麻法第 X 条，这个动作可能不符合 Y 规定，建议人工 review"
- AI **不能**说"这个动作合法/不合法"——最终判断需要持牌律师
- AI 系统本身需要真实大麻律师**年度审计**一次（~$2-5K/年）

**为什么 V0 优先**：
- 你 P5 Legal Library 已经有积累，起步快（6-8 周做出 MVP）
- 纯软件项目，不卡牌照下来
- 等许可证期间正好的事
- 关联 cowork 项目：`legal_library`

### 模块 5：顾客特征系统 ⭐ 自建
- 6 维度特征向量（口味 / 感觉 / 价格 / 形式 / 时段 / 耐受度）
- 去 PII 后存储（终极隐私安全）
- 服务于推荐 AI + 营销 AI 两个下游

### 模块 6：AI 营销系统（V2 / V3 阶段）
- 状态：可行性已确认（命格匹配 + 长期 ROI 高 + 合规可控）
- 不在 V1 阶段做，等顾客数据攒到一定规模再上
- 内容范围：邮件 / 短信文案生成 + 投放决策 + EDDM 卡片设计 + 复购触发
- 合规边界：所有 AI 生成内容必须人工审核才发出

### 模块 7：AI 会计师顾问（V1.5 阶段，2026-05-14 新增）
**大麻零售会计的真实痛点**：
- **IRC 280E**：联邦禁止大麻企业扣除运营费用（除了 COGS），有效税率高达 60-90%
- 每笔成本怎么分类（COGS vs 运营费用）= 直接影响税款几万到几十万
- 银行 / 支付受限（cannabis-friendly bank 才接）
- 多税种叠加（NY 大麻消费税 + 州销售税 + 联邦所得税 + 地方税）

**AI 会计师能做的事**：
- 实时分类（COGS vs 运营费用）→ 直接省 280E 税款
- 每日交易对账 / 现金流追踪 + 异常告警
- 月度财务报表自动生成
- 季度 / 年度 OCM 财务报表辅助
- 280E 优化建议

**核心边界**：
- AI 做 90% 日常记账，**报税 / 重大税务决策必须 CPA 签字**
- CPA 费用从 $30-50K/年 → $5-10K/年（AI 把工作量降 90%）

**与 AI 法律顾问的关系：AI 后台双柱**：
```
[操作发生]
   ↓
AI 法律顾问审"合不合规"
AI 会计师审"账目怎么处理"
   ↓
[操作通过 → 自动入账 + 合规标记]
```

**为什么放 V1.5 不放 V0**：
- 比 AI 法律顾问稍难（工程量 1.5-2 倍）
- 需要真实交易数据 + cannabis CPA 合作（牌照下来后才有）
- 你没有会计知识库积累（vs P5 法律积累）
- 顺序做：法律顾问跑通 → 再上会计师，节省 50% 精力

**V1 阶段过渡方案**：用现成 cannabis 会计软件（Cannapages / GreenGrowth）当基座，AI 增强层只做 280E 优化 + 实时分类两个最痛点。

## ⭐ SaaS 化路径（三阶段，2026-05-14 更新）

### 核心商业模式洞察

**这套系统从第一行代码开始，就要带着 SaaS 化的设计意图写。**
- ❌ "先做单店再想 SaaS" → 错
- ✅ "做 SaaS 但用单店当种子用户" → 对

### 数据网络效应（核心护城河）

```
更多店加入 → 更多数据 → 模型更准
→ 客户转化率更高 → 店家更不想离开
→ 口碑传播 → 更多店加入 ↻
```

- 单店 = 单店数据，模型质量受限
- 100 家店都用你的 SaaS = 100 家店数据汇总，质量碾压式领先

## ⭐ 供应链战略（2026-05-14 第三轮讨论确定）

### A. 大麻产品（Flower / Vape / Edibles）= 优先 Consignment 模式

**核心思路**：
- 优先争取 Consignment（代销）模式：品牌方先供货，卖出后结算
- 不强求 100%，**逐家谈判 + 混合模式**（部分 consignment + 部分 outright）
- 谈判筹码：你的"幽灵极速流"高换手率

**为什么**：
- 大麻零售最大压力 = 库存压货（一单 $5K-50K）
- Consignment = **0 现金流压力进货**
- 卖不出能退 = 风险转嫁给品牌方

### B. 辅件（1688 OEM 直采）= 4 种商业模式并行

**核心思路**：一种产品，4 种用法，每种都有明确商业目标

```
成本 $1 / 件（1688 OEM 直采）
   ↓
4 种使用场景，每种心理价值 $25：
```

**经济模型威力**：

- 内容：主公持有 SaaS 公司 / 公司 A 会用其服务 / 服务费市场公允

- 写入 Subscription Agreement
- 签字后 = 永远不能事后翻案

**满足这 3 条 = 100% 合法 self-dealing + 你永久安全**

## ⭐ 第 9 章：280E 税务策略（2026-05-25 整理）[ref-worthy]

> **关键事实**：联邦 IRC 280E 法案禁止大麻企业像普通企业那样从税前收入扣除运营费用（只能扣 COGS）。结果税基 ≈ 毛利润，有效税率高达 60-90%。**NY 州从 2023 年 1 月 1 日起跟联邦 280E 脱钩**——州税可正常扣 COGS + 运营开支。

### A. 联邦 vs NY 州（重要分裂）

| 维度 | 联邦 280E | NY 州（2023.1.1 起脱钩） |
|---|---|---|
| 可扣除项 | 仅 COGS（产品成本 + 部分直接生产费用） | COGS + 运营开支（房租 / 人工 / 营销 / 折旧 等） |
| 有效税基 | ≈ 毛利润 | ≈ 净利润（正常企业算法） |
| 税率 | 21% (C-Corp) / 37%+ (个人 passthrough) | NY 州正常税率 |

**P&L 例算**（$100K 月营收，~42% 毛利率，假设运营费 $30K）：
- 联邦：课税基础 ≈ $42K 毛利，21% C-Corp 联邦税 ≈ **$8.8K**
- NY 州：课税基础 ≈ $12K 净利润（$42K - $30K），正常税率 → 远低于联邦

### B. Entity 类型影响（C-Corp vs LLC，重要决策点）

| 类型 | 联邦税路径 | 280E 场景适用度 |
|---|---|---|
| **C-Corp** | 公司层 21% 联邦税 | 适合高 280E 税负，21% 比 37%+ 个人税率低 |
| **LLC（passthrough）** | 直接传给个人 37%+ 边际税率 | 高营收下比 C-Corp 更亏 |
| **S-Corp** | 介于两者之间 | 看股东个人税率情况 |

**当前 P12 计划**：Cannabis Retail LLC（详见第 8 章持股结构）—— ⚠️ **LLC 在 280E 场景下需要审慎评估**。

**待 CPA 咨询确认**：
1. 主公个人税率边际（决定 C-Corp vs LLC 谁更优）
2. 是否考虑 C-Corp + 工资发放策略（降低个人税）
3. 多 entity 拆分（公司 A 大麻店 LLC vs C-Corp / 公司 B Cowork Tech 不受 280E 影响）

### C. AI 增强应用（V1 方向）

第 4 章「功能模块清单」已写：AI 实时分类 COGS vs 运营费用 → 直接省 280E 税款。**但 NY 州 2023 脱钩后**，AI 分类的价值需要重新评估：
- 联邦层依然有用（COGS 分类影响联邦应纳税额）
- 州层不再敏感（运营费用州税都能扣）

### D. 来源 & 验证状态

## ⭐ 第 10 章：选址优先级公式（2026-05-25，一手经验验证）[ref-worthy]

> **核心公式**：居民收入水平 > 地铁节点 > 1km 内无竞争 > 停车（nice-to-have）

### A. 反直觉：人流密度 ≠ 盈利

**典型反例：法拉盛 Main St**
- 人流密度：极高
- 房租：$30-50K/月
- 实测预期 break-even 营收 ~$200K/月，**风险极高**
- 加上停车差 + 大客户不会在那买（社区文化原因）→ 不适合首店

### B. 一手经验验证（主公亲历）

**Bayside Cannabis（Bell Blvd + LIRR 站旁）**：
- 开门时间：Queens 不到 5 家牌照
- 房租：相对合理（具体待主公补）
- 实测：**日均 $1-2 万营收，快速起量**
- 成功要素拆解：
  1. **开门早** = 竞争少（不到 5 家）
  2. **地铁节点**（LIRR 站旁，通勤客流稳定）
  3. **居民收入不错**（Bayside 中产社区）
  4. ≠ 人流密度（Bell Blvd 不是法拉盛那种密度）

**Sage Seeds（Bellerose）**：
- 主公当前任职，一手日常运营经验
- 用途：作为选址公式的"对照参考"

### C. Queens 已饱和区（2025-2026 实测）

⚠️ **这些区域大概率不适合开新店**（已有覆盖，竞争密度高）：
- **Forest Hills**：The Flowery
- **Jamaica**：Silk Road
- **Ozone Park**：3-4 家
- **Kew Gardens**：A&P Dispensary
- **大部分中高收入区**：已有覆盖

→ 主公找地址时**优先避开**这些区，找尚未覆盖的中产社区 + 地铁节点。

### D. Sweet Spot 房租范围

- **目标**：$8K-$12K/月
- **Break-even 营收**：~$130-160K/月（可达）
- **风险位**：>$20K/月（如法拉盛），break-even ~$200K/月，**风险极高**

### E. 主公新地址评估流程（套这个公式）

主公给一个新地址 → 我按 4 维分析输出：
1. **居民收入水平**：查 zip code 中位数收入
2. **地铁/LIRR 节点距离**：步行 5min 内 = 加分
3. **1km 内竞争密度**：查 NY OCM 已发牌地图
4. **停车情况**：nice-to-have，不是决定因素
- 再叠加 **1000ft 禁区**（学校 / 教堂 / 公园等合规红线，详见「合规红线」章节）
- 输出：优点 + 风险 + 推荐度（高/中/低）

## ⭐ POS 谈判红线（Project Ghost Tech Stack）

> 这一章是 V0 阶段第一动作的 checklist——在写 AI 代码前必须完成 Flowhub 技术尽调

### 战略原则：Headless 架构
- **AI 是大脑**：负责体验、推荐、积分、定价、营销决策
- **POS 是手脚**：负责收银、Metrc 合规、扣减库存

### 6.1 核心 API 独裁能力（The Dictator API）

| 能力 | 重要性 | 说明 |
|------|--------|------|
| **订单注入 + 改价**（price_override / custom_discount_amount） | ⭐ 最关键 | AI 创建订单时直接指定最终成交价，POS 只执行不计算 |
| **身份"空投"合并**（Merge） | 必须 | 带手机号的预订单 + 扫驾照后自动合并法律信息（名字 / 生日 / 驾照号），无数据断层 |
| **测试沙盒** | 必须 | 签约前必须免费开发者 Sandbox |

### 6.2 运营 + 合规要求

- **实时库存 Webhook**：变动通知必须实时推送，**拒绝轮询**（polling）
- **BioTrack / Metrc 全自动同步**：拒绝人工手动上传表格
- **内置限购计算器**：超 NY 单次限购（3oz flower / 24g concentrate）自动锁单
- **iPad 支持**：拒绝绑定昂贵专有收银机
- **断网模式（Offline Mode）**：必须支持

### 6.3 商务条款 + 数据主权

- ⭐ **数据导出 = 死穴**：随时免费导出**完整客户名单 + 销售数据 CSV**，**必须包含手机号**，否则免谈
- **API / Webhook 包含在基础月费里**，不可用次数限制卡脖子

### Plan B 备用协议

如果 POS 厂商以"安全"为由拒绝 API 直接改价：
- **B1. 批量写入权限**：CSV 批量导入唯一优惠券码 / API 修改会员积分余额
- **B2. 购物车监听**：Cart Updated Webhook，服务器 2 秒内知道店员加了什么商品
