# 公司叙事追踪系统（防笔记库）— 方案定稿 2026-06-27

> **性质**：P9「主题累积研究 Loop」的**公司粒度分支**。不是新系统，复用同一套"假设→对答案"哲学。
> **由来**：2026-06-27 主公提"新闻追踪"，多轮对齐 + CC/Codex 两轮对抗审核后定稿。
> **作者**：CC（opus2）；对抗审核：Codex（gpt-5.5）
> **给未来 AI/主公看**：下次接手一看就懂这套要干啥、为什么这样设计、走到哪一步。

---

## 一、核心哲学（灵魂，不可动）

> **新闻不是资产，假设才是资产。不能改变某个投资假设状态的信息，不准进档案。**

档案的主角是**一组活的"假设"**（为什么这公司能赚钱），新闻只是**推动假设变化的燃料**。这条是整套系统防止沦为"勤奋新闻笔记库"的根。

判断它是不是笔记库的唯一标准：**它改变过决策没有**（含三级：改买卖动作 / 改观察级别 / 改触发条件）。改过=工具；从没改过=砍掉。

## 一·五、记录透明度铁律（2026-06-28 主公要求 + Codex 审核后立）

> **背景**：主公要求"每次往库里记东西，必须主动告知记什么+为什么"，防默默写库、防 AI 悄悄筛材料。
> **Codex 关键警告**：单人系统最大风险不是 AI 偷改信心，而是**AI 长期悄悄帮你筛材料，最后你以为看到全貌、其实是 AI 过滤后的叙事**。
> **判断线**（核心）：透明对象是"**语义动作**"（AI 有没有对材料做筛选/判断/归因），**不是**"数据库动作"（每行 insert）。不是只有"改信心"才重要——证据的选择、归因、方向，全是账本的一部分。

**三级报备：**
- 🔴 **动账本（逐条报"记什么+为什么"）**：新增/改/关假设、改信心、判应验/失效/部分、采纳/驳回 AI 建议、给证据定方向(支持/反对)。这些是改投资账本，一条都不能默默来。
- 🟡 **证据入库（让人看见、能拦截）**：选哪条新闻算证据、绑哪条假设、为什么重要。**当前 MVP 天然满足**——证据只能经手动 `narrative_dossier.py` 逐条录入（人在操作=天然可拦），哨兵**红线禁止自动入证据**。未来若做自动入库，必须改成"先列待入清单→人确认才入"。
- 🟢 **纯技术留痕（汇总报+声明未改判断）**：草稿存档、新闻原始快照、抓取日志。哨兵草稿底部固定带"记录状态"行，明确声明"仅留痕、未入证据库、未改任何假设"。

**否决项（守 YAGNI）**：Codex 提的统一"写库意图层(write_intent)"工程——MVP 阶段过度，用上面规则+习惯先跑，真觉得乱了再上。

## 二、为什么这样设计（取神去形）

- Codex 一审/二审给出完整重型方案（6表+日周月季四闸+档案预算+冷缓存+false_negative 回捞），哲学内核对，但落地过重。
- CC 判断：母系统地基假设（信号聚集能否预示机会）**8/24 才验证**，按"数据回来前不改策略 / YAGNI / 小步起步+数据驱动升级"纪律，**不能给未验证的假设建豪华工厂**。
- 三审收敛：CC 砍成轻量 MVP，Codex 认可砍法，只坚持补一个要害——**必须记录当时的操作判断**（否则证明不了它帮了决策）。
- 最终 = 取 Codex 的神（假设状态机 + 防笔记库硬标准），去其形（重型落地全部按需后置）。

## 三、最终设计（3 表 + 1 闸 + 1 可选）

### 3 张表

**表1 `narrative_hypotheses`（假设——档案主角）**
| 字段 | 说明 |
|---|---|
| id | 主键 |
| ticker | 股票代码 |
| statement | 假设内容（为什么能赚钱，hypothesis 语气：may/could） |
| trigger_condition | 触发条件（什么情况动作，带指标+阈值） |
| invalidation_condition | 失效条件（什么情况认错放弃） |
| confidence | 当前信心 1-5 |
| status | 成立中 / 被强化 / 被削弱 / 已失效 |
| importance | 核心 / 次要 / 仅观察（Codex 建议，防假设失控） |
| next_review_date | 下次必须复看日期（财报日等） |
| created_at / updated_at | 时间戳 |

**表2 `narrative_evidence`（证据——每条必须绑假设）**
| 字段 | 说明 |
|---|---|
| id | 主键 |
| hypothesis_id | 绑哪条假设（**绑不到 = 不存**） |
| direction | 支持 / 反对 |
| confidence_before / confidence_after | 把信心从 X 推到 Y |
| summary | 客观事实摘要（零主观，"营收X 同比Y"） |
| source / source_type | 来源 + 类型（SEC/PR/媒体/分析师，不同权） |
| event_date / logged_at | 事件日 / 入库时间 |

**表3 `narrative_weekly_checkins`（极简周记——证明帮了决策）**
| 字段 | 说明 |
|---|---|
| id | 主键 |
| ticker | 股票代码 |
| checkin_date | 日期 |
| top_hypothesis_change | 这周最大假设变化 |
| action_view | 当前操作倾向：买/持/加/减/避/等 |
| reason | 一句理由 |
| what_changes_my_mind | 下次什么会让我改主意 |

### 1 道闸（防笔记库唯一闸门）
新闻入库时问："这条推动了哪条假设？怎么推？" → 答不出 = **直接丢，不存**。

### 1 个可选 `narrative_discard_log`
每周最多 3 条"差点想存但绑不上假设"的新闻标题（只标题+日期，不研究）。用来监测"丢弃规则是不是太狠"。6-8 周若发现被丢的频繁变重要，再升级冷缓存。**可选，CC 倾向加（成本≈0）。**

## 四、怎么跑

- **财报日自动盯**（脚本，唯一自动部分）：yfinance 查财报日（记来源+抓取时间+允许更新，因为会变）→ 到点 Discord 提醒。
- **每周人肉**（CC）：扫该股新闻 → 能绑假设的入 evidence + 更新信心/状态 → 看有无假设触发/失效 → 填一行 weekly_checkin（含操作倾向）。
- **财报日复看（对答案）**：财报出来 → 对照假设的触发/失效条件 → 打勾（应验/没应验，结果分：命中/部分/未命中/无法验证/前提消失）+ 归因 + 更新信心。

⚠️ 先**全人肉**，不上自动 agent 链（等同母系统阶段0精神）。跑通了再考虑自动化。

---

## 五、两阶段验证（采 Codex）

- **阶段一**：VST 1 只 × 6-8 周 → 验"流程可不可用、我愿不愿维护、会不会退化成笔记、假设是否真被强化/削弱/失效"。
- **阶段二**（一阶段通过才做）：3 只 × 8-12 周 → 验"是否泛化"。
- ⚠️ **结论严格限定**为"这工作流值不值得扩展"，**不是**"系统是否有效/能赚钱"（1 只票样本不足以下后者结论）。

| | 主题累积研究 Loop（现有） | 本方案（公司分支） |
|---|---|---|
| 粒度 | 主题（AI电力） | 单公司（VST） |

### ⚠️ 边界铁律：建仓即移交（2026-06-27 主公追问"有无重复"后补）

实地核查现有 33 脚本 + 11 表：`thesis_monitor.py`（盯 Top3 持仓 thesis 成立/失效）、`dossier_autowrite.py`（趋势池价格轨迹）、`scanner_picks` 字段（建仓填一次/平仓判一次的 invalidation/bear_thesis/verdict）——**三者全围绕"已建仓持仓票"**。本方案补的是它们都没覆盖的空白：**建仓前 / 观察期对象的持续假设追踪 + 关注点演化 + 对答案**。

> 比喻：现有 = 已入职员工绩效考核；本方案 = 还在观察的候选人持续面试记录。是前置一环，不冲突。

**为防 4 套东西概念相似日后打架，定铁律**：
> 一个对象**一旦真建仓**，它的假设追踪即**移交给现有持仓监控体系**（thesis_monitor / scanner_picks），本方案只负责**建仓前的观察期**。不重复盯同一只持仓票。
> 实现：narrative_hypotheses 的 status 增加终态 `已移交持仓监控`；对象建仓后该档案归档，不再每周更新。
