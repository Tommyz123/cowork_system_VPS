# 主题累积研究 Loop — 设计存档 + 阶段路线图 + 验收标准

> **建立**：2026-06-23（主公 + BB 深度对话成型）
> **性质**：P9 第2层趋势主线的 theme_discovery 补环设计。给未来的 AI/主公看——下次接手一看就知道这套要干啥、走到哪一步、什么标准算过关。

## 一、要解决的问题（为什么做）

P9 现在天天采信号（trading.db `signals` 表，已 1710 条）但**只在"已选股"上用**——147 只有信号的股票里 132 只在持仓外，1376 条高质信号"堆在仓库没进选股视野"。

用 loop engineering 视角看：P9 是"金鱼记忆"——每季度扫描各扫各的，上季度对某主题的判断/踩的坑/验证了啥，下季度全忘重来。**缺的是第6零件「状态State」+ 第5零件「验证」**。

**目标**：给每个主题建一本"越写越厚、能查账"的研究档案，让系统从"采集器"进化成"会学习、会自我纠错的 loop"。

## 二、终态蓝图（三角色 + 三层档案）

**三角色（职责分离，避免又当运动员又当裁判）**：
1. **数据员（纯脚本，非AI）**：客观抓信号数/热度变化 → 写数据层。**零幻觉**。
2. **分析师 agent**：读数据层下判断 → 写分析层，每条挂数据依据。
3. **审计员 agent（独立）**：只干一件事——回看上轮判断 vs 这轮实际数据，打分"对/错/待定"。**不下新判断，只当裁判。**

> 为什么三角色：把"产生数据"从 AI 手里拿走给脚本，幻觉风险再降一层。判断者和验证者是不同 agent，堵死"两个乐观主义者点头"。

**三层档案（每主题一本，各层独立分区+时间戳）**：
```
主题：AI电力
【数据层】(脚本写,客观)   W1:信号5条 ORA,VST 热度+0% | W2:信号12条 +CEG 热度+140%
【分析层】(分析师agent写,挂依据) W1:"观望"←仅5条分散 | W2:"加速值得深挖"←翻倍+CEG进场
【审计层】(审计员agent写,只打分)  W2回看W1"观望"→对✅ | W3回看W2"加速"→待定🔍涨势放缓
```

**核心铁律**：数据层/分析层分离 = 主公 [[feedback_tracking_facts_only]] 铁律的延伸；验证用确定性规则（信号实际涨跌）判对错，不让 AI 自评。

## 三、⚠️ 地基假设（整套设计的命根，未验证）

**假设**："信号聚集 → 能预示主题机会"。

**这个假设从没被验证过。** 警示案例：VRRM 升温45倍最猛，但实为 Avis 合同终止暴跌-70% 的**坏消息扎堆**——信号多≠机会。所以好坏消息标记是必须的。

**所以不能一步到位搭整套**，必须先用最小成本验证地基，成立了再全建。否则=给错误前提建豪华工厂（loop engineering 最隐蔽的反模式）。

## 四、分阶段路线图（每阶段没验证就不进下一阶段）

### 阶段0：最小验证 / 排雷期 【← 当前在这】
- **做什么**：`theme_heatmap.py` 每周一自动出"信号升温榜"（升温倍数+🟢🔴好坏消息+持仓标记）发 Discord
- **谁判断**：BB（我）人肉下判断 + 人肉当审计员回看打分；主公每周扫一眼把关（1-2分钟）
- **目的**：验证地基假设——信号聚集到底能不能预示主题
- **不做**：不建子agent、不落库、不碰下单
- **已上线（2026-06-23）**：脚本 + cron 每周一 16:50 EDT + 验收提醒 cron（2026-08-24）

### 阶段1：半自动化（地基验证通过才做）
- 审计员 agent 化 + 三层档案落库（建表存判断+打分）
- 触发门槛：信号达 N 条、值得深挖的主题才启 agent 链；小主题纯脚本记录
- 仍是"AI判断+独立agent审计"，主公只看结果

### 阶段2：接入主线（阶段1稳了才做）
- 接进 theme_discovery，升温主题自动进选股候选视野
- 全自动 cron 化；主公只在"特别值得下手的主题"收提醒
- ⚠️ 红线：自动接选股 = 改策略本身，须等 P9 数据回来（呼应"数据回来前不改策略"）

## 五、验收标准（什么时候算"走得通"，可进下一步）

> 主公定的标准：数据自己说话，不是"我说有用就有用"。

**进入阶段1的 3 个条件（全满足才升级）**：

1. **时间够**：至少跑满 **6-8 周**（让"判断→下周打分"循环转过 6-8 轮）
2. **准确率达标**：BB 每周判断，下周回看 **命中率 ≥ 60%**
   - 标🟢"值得盯"的，下周真有 6成以上信号继续涨/股价涨 → 及格
   - 若跟瞎猜差不多（~50%）→ 信号聚集无预测力，**地基塌，不升级，停在这省功夫**
3. **主公认可**：数据达标后，把"6-8周命中率统计"摆给主公看，主公认可"确实有用" → 才进阶段1

**命中率怎么算**：每周标注的🟢/🔴判断 → 下周用 theme_heatmap 的新数据（信号是否继续升温）+ 该股实际涨跌核对 → 对=判断方向与下周实际一致，错=相反，待定=不明显。命中率 = 对 /（对+错）。

**验收时间点**：**2026-08-24 周一**（已设自动 Discord 提醒，到点叫主公，无需主公记着问）。

## 六.5、关联 BACKLOG：哨兵报告自动解读（🟡 缓做，2026-06-23 评估）

**想法**（主公提）：让自动告警（如 FERC 哨兵）不只发原始新闻，而是**自带 BB 的解读**（这是什么/重不重要/要不要动作），省掉"主公转发问 BB"那一步。

**定方案** = `B + 喂档案 + 回写档案`：哨兵触发时调 AI（claude --print）实时读新闻，**连相关档案（如趋势观察池 E1 历史判断）一起喂**，生成带上下文的解读（而非干巴复述），**解读完回写档案**形成累积。淘汰 A（固定模板）——哨兵报的是新事件，模板必过时。

**为什么缓做（ROI 判断）**：FERC 哨兵盯终局裁决，一年才触发 1-2 次；人肉解读 2 分钟，非高频痛点。为低频事件搭"自动调 AI"链路 ROI 低（呼应 [[feedback_yagni]]）。

**触发条件（什么时候做）**：FERC 8-9 月终局裁决那次 / 或哨兵报警变频繁 / 或有多个哨兵都需自动解读时 → 那时这套才真正派上用场。
